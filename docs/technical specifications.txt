Экосистема для аквариумов/террариумов
Контроллер + web/desktop приложение

Актуализация на 29 апреля 2026:
- Текущий backend уже имеет микросервисный каркас: Gateway, Identity, Device, Telemetry, Control, Notification.
- Telemetry ingress реализован через DeviceService: POST /api/device/v1/sensors/telemetry.
- Для первого реального МК появился direct device-api lab mode: http://localhost:5237.
- Gateway path остается защищенным вариантом для клиентов: JWT + X-Device-Token.
- CreatedAtRoute регистрации контроллера исправлен.
- Пустой batch и Guid.Empty sensor id теперь валидируются.
- Batch-size validator принимает 1..50 элементов.
- Telemetry validation имеет явные messages для MacAddress, Items, SensorId, ExternalMessageId, RecordedAt; ExternalMessageId ограничен 100 символами.
- value: 0 больше не отклоняется validator-ом DeviceService.
- Unknown/unowned sensor ids возвращаются в validationErrors вместе со skippedCount.
- Авторизация protected API работает через Authorization: Bearer и AccessToken cookie fallback.
- IdentityService получил обязательный TimeZone и миграцию AddTimeZoneForUser; в register DTO поле пока называется TimaZone.
- DeviceService и TelemetryService собираются через dotnet build ... -m:1 без ошибок.
- Relay command enqueue path уже использует AddAsync.
- Relay command polling помечает команды как Sent и повторно выдает зависшие Sent команды через 1 минуту, максимум 3 попытки, до ExpireAt.
- Unique IsAggregated regression закрыт в telemetry config/snapshot/migration 20260429103915; миграции применены, runtime-БД считается обновленной на non-unique indexes.
- Перед стабильным подключением МК нужно замкнуть Ecosystem projection для TelemetryService, исправить CreateEcosystemAsync existing/null bug, подтвердить telemetry validation для прошивки runtime smoke-ом, подтвердить compose/E2E smoke и проверить alert chain.

Основные функции:
- добавление нескольких аквариумов/террариумов;
- удаленное управление;
- контроль состояния;
- аналитика;
- уведомления при отклонениях;
- калькуляторы и вспомогательные расчеты.

Исполнительные устройства:
- реле 1: фильтр;
- реле 2: фильтр;
- реле 3: нагреватель;
- реле 4: насос;
- реле 5: лампа;
- реле 6: свет;
- реле 7-8: резерв под расширение.

Датчики:
- XKC-Y25-V как бесконтактный датчик уровня воды;
- I2C soil moisture sensor как датчик влажности грунта/субстрата;
- AHT20 I2C как датчик влажности и температуры поверхности/окружения;
- DS18B20 как погружной датчик температуры воды;
- второй DS18B20 или отдельный адрес на OneWire-шине для температуры поверхности;
- мутность воды позже;
- потенциально pH/kH/NO3 как ручные или аппаратные измерения позже.

Локальный UI первого МК:
- LCD1602 для статуса;
- светодиоды для состояния;
- энкодер KY-040 и обычная кнопка-замыкалка для навигации;
- семисегментный индикатор 5641AS не используется;
- потенциометры для навигации не используются.

Регулирование:
- температура воды;
- температура поверхности;
- свет;
- включение/выключение по расписанию;
- плавное затухание/рассвет/закат;
- количество и частота кормления.

Режим отпуск:
- пользователь задает время отсутствия;
- система оценивает расходники для автономной работы;
- позже можно добавить сценарий работы от аккумуляторов.

Журнал обслуживания:
- цифровой паспорт аквариума/террариума;
- напоминания о подмене воды, замене ламп, обслуживании фильтров;
- ручная запись тестов воды и графики.

Контроль тока/энергомониторинг:
- возможные чипы: INA219, PZEM-004T или аналог;
- детекция поломок: реле включено, а потребление 0 Вт;
- alert пользователю при подозрении на отказ насоса/нагревателя/лампы.

Planned архитектурные расширения:

1. `planned`: SignalR live dashboard
- Gateway или отдельный realtime edge подписывается на telemetry/alert events.
- UI получает новые значения без polling.

2. `planned`: gRPC внутренняя синхронная связь
- Пример: ControlService проверяет лимиты подписки в IdentityService перед созданием нового правила.
- Использовать только там, где event-driven модели недостаточно.

3. `planned`: MediatR + CQRS
- Начать с ControlService.
- Разнести команды и чтение по handler-ам.
- Добавить pipeline behaviors для валидации, логирования и idempotency.

4. `planned`: CQRS + Read Model
- Для аналитики и агрегатов.
- Пример: DailyStats для быстрых графиков за месяц/год без тяжелых запросов по telemetry history.

5. `planned`: gRPC streaming
- Эмулятор или МК держит постоянное соединение с DeviceService.
- Сервер может push-ить команды и конфигурацию без ожидания polling.

Рекомендуемый порядок:
1. Подтвердить docker compose и E2E smoke.
2. Исправить telemetry persistence blocker: Ecosystem projection.
3. Подтвердить validation contract для МК на running stack.
4. Подключить МК/эмулятор через direct device-api lab mode.
5. Подтвердить telemetry -> control -> relay -> notification.
6. Только после этого начинать CQRS/MediatR, SignalR и gRPC.
