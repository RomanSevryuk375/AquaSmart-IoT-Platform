## Критичные текущие замечания

### 1. Build использует зафиксированный последовательный режим

Файл:
- `global.json`

Проблема:
- корневой `global.json` теперь валиден и выбирает .NET 8 SDK;
- `dotnet build` для `DeviceService` и `TelemetryService` подтвержден через `-m:1` на 29 апреля 2026;
- параллельный restore/build без `-m:1` может падать без явных compile errors;
- без writable `DOTNET_CLI_HOME` в sandbox/CI-like окружении `dotnet` может падать на first-time setup.

Статус:
- последовательная команда сборки уже описана в документации;
- tooling follow-up остается только для расследования параллельного restore/build без `-m:1`.

### 3. Ecosystem sync для TelemetryService не замкнут

Файлы:
- `src/Contracts/Contracts/Events/EcosystemEvents/EcosystemCreatedEvent.cs`
- `src/Services/TelemetryService/Telemetry.Application/Services/TelemetryDataService.cs`
- `src/Services/TelemetryService/Telemetry.Application/Services/SensorService.cs`
- `src/Services/ControlService/ControlService.Application/Services/AquariumService.cs`

Проблема:
- `TelemetryService` ищет `Ecosystem` по `ControllerId` перед сохранением telemetry;
- consumer `EcosystemCreatedEvent` есть;
- publisher `EcosystemCreatedEvent` в текущем коде не найден;
- `ControlService` публикует aquarium events (`AquariumCreatedEvend`), а не ecosystem events.
- `EcosystemService.CreateEcosystemAsync` сейчас содержит инвертированную проверку: при `existingEcosystem is null` возвращает `Success` без создания projection, а при существующей записи пытается создать дубль.

Почему это плохо:
- telemetry batch может быть принят `DeviceService`, но consumer в `TelemetryService` вернет retryable error `Ecosystem for controller ... not found`;
- sensor projection в `TelemetryService` также зависит от существующего ecosystem.

### 4. Text ошибок telemetry validation требует чистки

Файлы:

Почему это плохо:

### 6. Auth contract требует production hardening

Файлы:
- `src/Services/IdentityService/IdentityService/Controllers/AuthController.cs`
- `src/Contracts/Contracts/Authorization/Extensions.cs`
- `src/Gateway/ApiGateway/ApiAuthentication.cs`

Статус:
- login/register/refresh выставляют `AccessToken` и `RefreshToken` cookies;
- gateway и сервисы используют общий `JwtBearer` setup, который читает `Authorization: Bearer <accessToken>` и fallback `AccessToken` cookie;
- Bearer header остается основным API-контрактом, cookie fallback задокументирован для browser clients;
- MCU endpoints через gateway блокируются route-level policy `/api/device/*` до попадания в `[AllowAnonymous]`, поэтому первый МК остается на direct `device-api:5237` + `X-Device-Token`.

Оставшееся:
- production HTTPS требует отдельной cookie/security policy.

## Минимальный стандарт перед следующим ревью

1. Синхронизация `Ecosystem` для `TelemetryService` замкнута producer/consumer-ом, включая fix `EcosystemService.CreateEcosystemAsync`.
2. Direct MCU lab mode подтвержден runtime-smoke-ом.
3. Validation errors для telemetry ingress подтверждены smoke-ом: `ExternalMessageId`, `value: 0`, `RecordedAt`, `items`, batch size, unknown sensor ids.
4. Relay command flow подтвержден smoke-ом, включая retry `Sent` команды после потерянного ack.
5. Пройден `docker compose up --build`.
6. Пройден smoke: register/login -> create controller/sensor -> aquarium -> telemetry -> telemetry saved -> control reaction.
7. Refresh token reuse/family revocation policy описана или реализована.
