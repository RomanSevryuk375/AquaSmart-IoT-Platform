## Обзор архитектуры

Бэкенд построен как распределенная система с использованием **Микросервисной архитектуры**. Проект спроектирован с учетом высокой доступности, отказоустойчивости и строгого разделения ответственности (Separation of Concerns).

### Состав микросервисов
*   **API Gateway (YARP)**: Единая точка входа. Отвечает за маршрутизацию запросов, терминацию SSL и централизованную проверку JWT.
*   **Identity Service**: Управление пользователями, аутентификация, ротация refresh-токенов и управление доступом на основе подписок (RBAC).
*   **Device Service**: Регистрация оборудования, синхронизация конфигураций и формирование очереди команд для реле. Выступает в роли gRPC-сервера для валидации оборудования.
*   **Control Service**: Вычисление правил автоматизации, обработка расписаний (Cron через Quartz) и управление режимом отпуска.
*   **Telemetry Service**: Прием высокочастотных данных с датчиков, агрегация данных для оптимизации хранения в БД и трансляция метрик в реальном времени через SignalR.
*   **Notification Service**: Обработка системных событий и асинхронная доставка уведомлений через Telegram API и SMTP.

## Межсервисное взаимодействие

Для обеспечения слабой связанности (decoupling) и высокой производительности применяется гибридная стратегия коммуникации:

1.  **Асинхронный обмен сообщениями (RabbitMQ & MassTransit)**
    Используется для обеспечения согласованности в конечном счете (eventual consistency). Для решения проблемы двойной записи (dual-write problem) реализован паттерн **Transactional Outbox**. Доменные события сериализуются в таблицу `outbox_messages` в рамках той же транзакции EF Core, а затем отправляются фоновым воркером Quartz.
2.  **Синхронный RPC (gRPC)**
    Применяется исключительно для строгих высокопроизводительных внутренних проверок, где ожидание консистентности недопустимо. Контракты определены через `.proto` файлы (например, `DeviceIntegrationGrpc`). Микросервисы Control и Notification опрашивают Device Service по gRPC для подтверждения прав на управление оборудованием.
3.  **Потоковая передача данных (SignalR)**
    Используется в Telemetry Service для трансляции сырых и агрегированных данных датчиков подключенным веб- или мобильным клиентам.

## Стратегия тестирования

Платформа включает строгий многоуровневый пайплайн автоматизированного тестирования, управляемый через **GitHub Actions** (`microservices-ci.yml`).

<details>
<summary><b>Раскрыть детали уровней тестирования</b></summary>

*   **Архитектурные тесты (`NetArchTest.Rules`)**
    Гарантируют строгое соблюдение Clean Architecture. Тесты проверяют, что слой Domain не имеет внешних зависимостей, слой Application зависит только от Domain, а классы обработчиков и валидаторов соответствуют конвенциям именования.
*   **Unit-тесты (`xUnit`, `NSubstitute`, `FluentAssertions`)**
    Изолированное тестирование доменных сущностей, Value Objects, поведений MediatR (например, `TransactionBehavior`) и бизнес-логики без обращения к базе данных.
*   **Интеграционные тесты (`Testcontainers`, `Respawn`)**
    Запускают эфемерные контейнеры PostgreSQL для каждого прогона тестов. Проверяют маппинги EF Core, SQL-запросы Dapper и обработчики MediatR при работе с реальной БД. Пакет `Respawn` используется для чистой очистки состояния БД между тестами.
*   **End-to-End (E2E) тесты (`WebApplicationFactory`)**
    Проверяют полный жизненный цикл HTTP-запроса, включая маршрутизацию API Gateway и авторизацию. Мокированные обработчики аутентификации обходят фактическую генерацию токенов для фокусировки на логике эндпоинтов.
</details>

## Применяемые паттерны и принципы

Кодовая база строго следует принципам **SOLID** и использует современные паттерны проектирования корпоративных систем.

### 1. Domain-Driven Design (DDD)
*   **Rich Domain Models**: Сущности (например, `Sensor`, `Relay`) инкапсулируют собственное состояние. Мутация состояния возможна только через явные доменные методы, защищающие инварианты.
*   **Value Objects**: Концепции вроде `MacAddress`, `ConnectionAddress` и `ConditionThreshold` смоделированы как неизменяемые записи (records) со встроенной валидацией.

### 2. CQRS (Command Query Responsibility Segregation)
*   **Commands (Запись)**: Обрабатываются через Entity Framework Core 8 для использования механизма отслеживания изменений (change tracking) и сложной доменной логики.
*   **Queries (Чтение)**: Обрабатываются через **Dapper**. Сырые SQL-запросы обходят накладные расходы EF Core, напрямую проецируя представления БД в легковесные DTO.
*   **Pipeline Behaviors**: Сквозной функционал (логирование, валидация через `FluentValidation`, транзакции и проверка прав доступа) внедрен через поведения MediatR.

### 3. Паттерн Result (Railway-Oriented Programming)
Исключения (Exceptions) не используются для управления потоком бизнес-логики. Вместо этого применяется кастомная реализация паттерна `Result<T>` и `Error`, которая транслирует строго типизированные доменные ошибки в стандартизированные HTTP-ответы.

## Архитектура встроенных систем (ESP32 / C++)

Прошивка для микроконтроллера ESP32 спроектирована с учетом отказоустойчивости в условиях нестабильных сетей.

*   **Конечный автомат (FSM)**: Базовая логика работает как конечный автомат (`BOOT`, `CONNECT_WIFI`, `ONLINE`, `DEGRADED_OFFLINE`, `SAFE_MODE`).
*   **Офлайн-устойчивость (Circular Buffer)**: Реализована кастомная очередь `TelemetryQueue`, аллоцированная в IRAM. При потере сети данные буферизуются локально. При восстановлении связи весь пакет (batch) отправляется на сервер.
*   **Аппаратная безопасность**: Критические показатели (например, датчик уровня воды) вызывают аппаратные прерывания для мгновенного отключения высоковольтных реле в обход бэкенда для обеспечения физической безопасности.

## Локальное развертывание

Вся инфраструктура бэкенда полностью контейнеризирована и может быть запущена одной командой.

**Требования**: Docker Desktop

Склонируйте репозиторий:
   ```bash
   git clone https://github.com/romansevryuk375/aquasmart-iot-platform.git
   cd aquasmart-iot-platform
   ```
Запустите инфраструктуру (БД, RabbitMQ, стек ELK, микросервисы):
    ```
    docker compose up --build -d
    ```
Доступ к конечным точкам:

        API Gateway: http://localhost:5055

        RabbitMQ Management: http://localhost:15672 (guest/guest)

        Kibana (Логи): http://localhost:5601

Миграции Entity Framework применяются автоматически при старте контейнеров с помощью DatabaseMigrationService.
