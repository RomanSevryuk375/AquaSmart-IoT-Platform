## Обзор архитектуры

Серверная часть построена как распределенная система с использованием **микросервисной архитектуры** (Microservices Architecture). Проект спроектирован с упором на высокую доступность, отказоустойчивость и строгое разделение ответственности (separation of concerns).

### Состав микросервисов
*   **API Gateway (YARP)**: Выступает в роли единой точки входа. Отвечает за маршрутизацию запросов, терминацию SSL и централизованную валидацию JWT.
*   **Identity Service**: Управляет пользователями, аутентификацией через JWT, ротацией refresh-токенов и контролем доступа на основе подписок/ролей (RBAC).
*   **Device Service**: Отвечает за инициализацию (provisioning) аппаратных узлов, синхронизацию конфигураций и управление очередями команд для реле.
*   **Control Service**: Выполняет оценку правил автоматизации, управление расписаниями (на базе Cron через Quartz) и режимами отпуска.
*   **Telemetry Service**: Принимает высокочастотные данные с датчиков, агрегирует их для оптимизации хранения в БД и рассылает обновления в реальном времени через SignalR.
*   **Notification Service**: Обрабатывает оповещения и обеспечивает асинхронную доставку сообщений через Telegram API и SMTP.

## Применяемые паттерны и принципы

Кодовая база строго следует принципам **SOLID** и активно использует современные паттерны проектирования корпоративных систем.

### 1. Предметно-ориентированное проектирование (DDD)
*   **Богатые доменные модели (Rich Domain Models)**: Сущности (например, `SensorEntity`, `RelayEntity`) инкапсулируют собственное состояние. Мутация состояния возможна только через явные доменные методы, что защищает инварианты системы.
*   **Корни агрегации (Aggregate Roots)**: Реализованы базовые классы `AggregateRoot`, отслеживающие доменные события (`IDomainEvent`).
*   **Доменные события (Domain Events)**: Изменения состояния генерируют события (например, `SensorNoDataDomainEvent`), которые затем маппятся в интеграционные события для межсервисного взаимодействия.

### 2. CQRS (Command Query Responsibility Segregation)
*   Реализовано с использованием библиотеки **MediatR**. 
*   Операции чтения (Queries) и записи (Commands) обрабатываются строго изолированно.
*   **MediatR Pipeline Behaviors**: Сквозной функционал (cross-cutting concerns) вынесен в пайплайны:
    *   `ValidationBehavior`: Интегрирован с `FluentValidation` для прерывания некорректных запросов.
    *   `TransactionBehavior`: Автоматически оборачивает выполнение команд в транзакции базы данных EF Core.
    *   `SecurityBehavior`: Обеспечивает изоляцию данных между арендаторами (tenant isolation), гарантируя, что пользователи могут изменять только собственные аппаратные сущности.

### 3. Паттерн Transactional Outbox
Для гарантии итоговой согласованности (eventual consistency) между микросервисами и решения проблемы двойной записи (dual-write problem) нативно реализован паттерн Outbox:
*   Перехватчик EF Core `SaveChangesInterceptor` (`ConvertDomainEventsToOutboxMessagesInterceptor`) отлавливает доменные события непосредственно перед коммитом транзакции, сериализует их и сохраняет в таблицу `outbox_messages` в рамках той же транзакции.
*   Отказоустойчивый фоновый процесс **Quartz.NET** непрерывно опрашивает таблицу Outbox и публикует сообщения в **RabbitMQ** через **MassTransit**, гарантируя семантику доставки *at-least-once*.

### 4. Паттерн Result (Railway-Oriented Programming)
Исключения (Exceptions) не используются для управления потоком бизнес-логики. Вместо этого на всех слоях применяется кастомный паттерн `Result<T>` и `Error`.
*   Доменный слой возвращает строго типизированные ошибки (например, `Error.Validation`, `Error.NotFound`).
*   Метод расширения (`ToActionResult`) на слое представления автоматически конвертирует доменные ошибки в соответствующие HTTP статус-коды (400, 404, 409), сохраняя контроллеры максимально тонкими.

### 5. Паттерны проектирования GoF
*   **Паттерн Стратегия (Strategy)**: Используется в `TelemetryService` для оценки динамических правил автоматизации (`EqualConditionEvaluator`, `GreaterConditionEvaluator`).
*   **Фабричный метод (Factory Method)**: Фабрики `RuleEvaluatorFactory` и `RuleOperatorFactory` инстанцируют нужные стратегии оценки на основе пользовательских конфигураций.
*   **Репозиторий и Единица работы (Repository & Unit of Work)**: Абстрагируют `DbContext` из EF Core, предоставляя унифицированный интерфейс для доступа к данным и управления транзакциями.

## Архитектура встроенных систем (ESP32 / C++)

Прошивка для микроконтроллера ESP32 спроектирована с учетом отказоустойчивости в нестабильных сетевых условиях.

*   **Конечный автомат (Finite State Machine)**: Базовая логика работает на основе конечного автомата (`SystemState::BOOT`, `CONNECT_WIFI`, `ONLINE`, `DEGRADED_OFFLINE`, `SAFE_MODE`).
*   **Офлайн-устойчивость (Кольцевой буфер)**: Реализована кастомная структура `TelemetryQueue`, размещенная в IRAM. При сбое Wi-Fi или недоступности API узел продолжает сбор данных с датчиков и переходит в состояние `DEGRADED_OFFLINE`. При восстановлении связи накопленная телеметрия отправляется на сервер батчами.
*   **Аппаратные системы безопасности (Fail-Safes)**: Превышение критических порогов (например, датчиков уровня воды) вызывает жестко закодированные аппаратные прерывания. Микроконтроллер локально отключает высоковольтные реле, полностью игнорируя бэкенд для гарантии физической безопасности.

## Локальное развертывание

Вся серверная инфраструктура контейнеризирована и может быть запущена одной командой.

**Требования**: [Docker Desktop](https://www.docker.com/products/docker-desktop/)

1. Клонируйте репозиторий:
   ```bash
   git clone https://github.com/ВашПользователь/romansevryuk375-aquaapi.git
   cd romansevryuk375-aquaapi
   ```

2. Запустите инфраструктуру (Базы данных, RabbitMQ, Микросервисы):
   ```bash
   docker compose up --build -d
   ```
Точки доступа: API Gateway: http://localhost:5055

Панель управления RabbitMQ: http://localhost:15672 (Учетные данные: guest/guest)

Примечание: Миграции Entity Framework применяются автоматически при запуске контейнеров. Дождитесь прохождения всех проверок работоспособности (Healthchecks) в течение 15-30 секунд.
